## Q1. 프로세스의 PPID 확인

**문제:** 프로세스의 PPID를 확인하는 올바른 명령어는?

| 선택지 | 명령어 |
|--------|---------|
| A | `ps` |
| B | `ps -e` |
| C | `ps a` |
| D | `ps -f` |

**정답:** D. `ps -f`

**해설:** `ps -f` 명령어는 프로세스를 전체 형식으로 표시하며, UID, PID, C, STIME, TTY, TIME, CMD와 함께 PPID(부모 프로세스 ID) 열을 포함한다. 기본 `ps` 명령어는 PID, TTY, TIME, CMD만 표시하며 PPID 정보는 포함하지 않는다.

## Q2. 특정 사용자의 프로세스 확인

**문제:** 사용자 `han1`과 관련된 프로세스 정보를 확인하는 데 **올바르지 않은** 명령어는?

| 선택지 | 명령어 |
|--------|---------|
| A | `ps -u han1` |
| B | `ps -fu han1` |
| C | `ps -p $(pgrep -u han1)` |
| D | `ps -pu han1` |

**정답:** D. `ps -pu han1`

**해설:** 선택지 A, B, C는 모두 `han1` 소유의 프로세스를 나열하는 유효한 방법이다. `-u` 옵션은 사용자별로 필터링하고, `-fu`는 전체 형식과 사용자 필터를 결합하며, `-p`와 `pgrep`을 사용하면 PID로 프로세스를 선택한다. 그러나 `ps -pu han1`은 `-p`가 사용자 이름이 아닌 PID 목록을 기대하므로 올바르지 않다.

## Q3. crontab 스케줄링

**문제:** 12월 30일 14시 30분에 `/tmp` 디렉터리의 파일 목록을 `tmp.out`에 저장하는 작업을 예약하려면, 어떤 crontab 항목이 올바른가?

| 선택지 | crontab 항목 |
|--------|---------------|
| A | `12 30 14 30 * /usr/bin/ls -l /tmp > ~user1/tmp.out` |
| B | `30 14 30 12 * /usr/bin/ls -l /tmp > ~user1/tmp.out` |
| C | `14 30 12 30 * /usr/bin/ls -l /tmp > ~user1/tmp.out` |
| D | `14 30 30 12 * /usr/bin/ls -l /tmp > ~user1/tmp.out` |

**정답:** B. `30 14 30 12 * /usr/bin/ls -l /tmp > ~user1/tmp.out`

**해설:** crontab 형식은 `분 시 일 월 요일 명령어`이다. 12월 30일 14시 30분의 경우: 분=30, 시=14, 일=30, 월=12, 요일=*(모든 요일). 따라서 올바른 항목은 `30 14 30 12 *`이다.

## Q4. 프로세스 설명

**문제:** 다음 프로세스에 대한 설명 중 **올바르지 않은** 것은?

| 선택지 | 설명 |
|--------|-------------|
| A | 프로세스는 현재 실행 중인 프로그램이다. |
| B | 프로세스는 부모-자식 관계를 가진다. |
| C | 프로세스는 PID 번호를 가진다. |
| D | 프로세스는 UID 번호를 가진다. |

**정답:** D. 프로세스는 UID 번호를 가진다.

**해설:** 프로세스는 어떤 사용자가 소유하는지 식별하는 UID(사용자 ID)와 연관되어 있지만, UID는 프로세스 자체의 고유 속성이 아니라 사용자에게 속한다. 프로세스는 자체적인 PID(프로세스 ID)를 가지며 부모로부터 PPID(부모 PID)를 상속받는다. 프로세스가 PID를 "가지는" 것과 같은 방식으로 UID를 "가지는" 것이 아니라, UID는 프로세스를 실행한 사용자의 속성이다.

## Q5. `at` 명령어 사용 조건

**문제:** 사용자 계정이 `at` 명령어를 사용하기 위한 **유효하지 않은** 조건은?

| 선택지 | 조건 |
|--------|-----------|
| A | `/etc/at.allow`가 존재하고 사용자 계정이 여기에 나열되어 있다. |
| B | `/etc/at.deny`만 존재하고 사용자 계정이 여기에 나열되어 있지 않다. |
| C | `/etc/at.allow`와 `/etc/at.deny`가 모두 존재하고 사용자 계정이 둘 다에 나열되어 있다. |
| D | `/etc/cron.allow`와 `/etc/cron.deny`가 모두 존재하지 않는다. |

**정답:** D. `/etc/cron.allow`와 `/etc/cron.deny`가 모두 존재하지 않는다.

**해설:** `at` 명령어의 접근 제어는 `cron` 파일이 아닌 `/etc/at.allow`와 `/etc/at.deny`를 사용한다. `/etc/at.allow`가 존재하면 여기에 나열된 사용자만 `at`을 사용할 수 있다. `/etc/at.deny`만 존재하면 여기에 나열되지 않은 사용자가 `at`을 사용할 수 있다. 둘 다 존재하면 `/etc/at.allow`가 우선한다. `cron.allow`와 `cron.deny` 파일은 `at` 명령어 접근과 무관하다.

## Q6. 사용자 이름으로 프로세스 찾기

**문제:** `guest`라는 사용자가 실행한 모든 프로세스를 찾는 명령어는?

**정답:** `ps -u guest`

**해설:** `ps -u` 옵션은 유효 사용자 ID 또는 이름으로 프로세스를 필터링한다. `ps -u guest`를 실행하면 `guest` 사용자 계정에서 현재 실행 중인 모든 프로세스가 표시된다.

## Q7. 프로세스 강제 종료

**문제:** PID 5000인 프로세스가 `kill 5000`으로 종료되지 않는다. 이 프로세스를 강제로 종료하려면 어떻게 해야 하는가?

**정답:** `kill -9 5000`

**해설:** 기본 `kill` 명령어는 SIGTERM(시그널 15)을 전송하며, 프로세스가 종료 전에 정리 작업을 수행할 수 있게 한다. 프로세스가 SIGTERM을 무시하면 `kill -9`(SIGKILL)을 사용하여 강제로 종료할 수 있다. SIGKILL은 프로세스가 포착하거나 무시할 수 없어 종료가 보장된다.

## Q8. 백그라운드 작업 종료

**문제:** 아래와 같이 세 개의 백그라운드 작업이 있을 때, 작업 번호 3을 종료하는 명령어를 작성하라:
```
$ jobs
[1] + Running    sleep 100
[2] - Running    find / -name test
[3]   Running    sleep 300
```

**정답:** `kill %3`

**해설:** bash에서 백그라운드 작업은 `%` 뒤에 작업 번호를 붙여 참조한다. `kill %3`은 작업 번호 3에 SIGTERM을 전송한다. 이는 PID 3인 프로세스를 종료하려는 `kill 3`과 다르다.

## Q9. 매주 반복 crontab 설정

**문제:** 매주 일요일 오후 11시에 사용자 `user01`의 프로세스 목록을 확인하여 사용자 홈 디렉터리의 `ps.out`에 저장하는 crontab 항목을 설정하라.

**정답:** `0 23 * * 0 /bin/ps -u user01 > /home/user01/ps.out`

**해설:** crontab 형식은 `분 시 일 월 요일 명령어`이다. 매주 일요일 오후 11시의 경우: 분=0, 시=23, 일=*(모든 일), 월=*(모든 월), 요일=0(일요일). `ps -u user01` 명령어는 `user01` 소유의 프로세스를 나열하며, 출력이 지정된 파일로 리다이렉트된다.

## Q10. 예약된 `at` 작업 삭제

**문제:** 다음과 같이 예약된 `at` 작업이 있을 때, 첫 번째 작업을 삭제하는 명령어는?
```
$ atq
8  Sun May 31 15:30:00 2020  a  user1
9  Sun May 31 16:00:00 2020  a  user1
```

**정답:** `atrm 8`

**해설:** `atrm` 명령어는 작업 번호로 예약된 `at` 작업을 삭제한다. `atq`는 대기 중인 작업을 ID와 함께 나열하며, `atrm 8`은 작업 번호 8을 삭제한다. 또는 `at -d 8`을 사용하여 작업을 삭제할 수도 있다.

## Q11. 모든 사용자에게 `at` 명령어 허용

**문제:** 모든 사용자가 `at` 명령어를 사용할 수 있도록 Ubuntu Linux를 설정하려면 어떻게 해야 하는가?

**정답:** `/etc/at.allow` 파일을 삭제하고(또는 존재하지 않도록 하고), `/etc/at.deny` 파일이 비어 있거나 존재하지 않도록 한다.

**해설:** `at` 접근 제어 로직은 다음과 같이 동작한다: `/etc/at.allow`가 존재하면 여기에 나열된 사용자만 `at`을 사용할 수 있다. `/etc/at.allow`가 존재하지 않고 `/etc/at.deny`가 비어 있거나 없으면 모든 사용자가 `at` 명령어를 사용할 수 있다. `/etc/at.allow`를 제거하고 `/etc/at.deny`를 비워 두면 어떤 사용자도 접근이 거부되지 않는다.
